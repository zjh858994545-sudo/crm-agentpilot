# CRM-AgentPilot 学习版讲解稿

这份文档不是给代码审查用的，而是给你自己理解产品、复盘设计和对外讲解用的。核心目标是：你不需要把每个实现细节都背下来，但要能讲清楚“这个产品解决什么业务问题、页面怎么用、后端怎么支撑、AI 在哪里发挥作用、哪些地方是商业化演进项”。

## 1. 一句话理解这个项目

CRM-AgentPilot 是一个面向本地生活销售场景的 CRM AI Agent 工作台。

它不是普通聊天机器人，而是把客户、商机、销售知识库、AI 建议、CRM 写入确认、运行审计串成一条销售作业流：

客户数据 -> 商机排序 -> Agent 分析/建议 -> 写操作确认 -> CRM 落库 -> 运行审计与评测。

对外讲解时可以这样说：

> 我做的是一个 CRM 销售作业 Agent。它的重点不是让模型随便聊天，而是让 Agent 能安全进入 CRM 流程：读客户和商机数据，调用知识库，给销售建议；但凡要写 CRM，比如创建任务、写联系记录、更新商机阶段，都必须先生成确认记录，用户确认后才真正落库。

## 2. 三个角色怎么理解

### 销售

销售每天最关心三件事：

- 今天先跟进谁
- 跟进时怎么说
- 跟进完怎么把任务或联系记录写回 CRM

对应页面：

- 今日工作台
- 客户 360
- Agent 工作台
- 知识库问答
- 呼叫中心

### 销售主管

主管更关心团队优先级和商机质量：

- 哪些商机价值高
- 哪些客户风险高
- 推荐理由是否可解释
- 团队跟进是否有节奏

对应页面：

- 商机优先级

### 系统管理员

管理员不直接做销售动作，而是看系统是否可靠：

- Agent 每次运行了什么
- 调了哪些工具
- 写操作有没有经过确认
- RAG 和 Tool Calling 评测结果怎么样

对应页面：

- 系统能力
- 运行审计
- 质量评估

## 3. 完整业务链路怎么讲

推荐用这条线讲，不要从技术栈开始讲：

1. 销售打开今日工作台，看系统推荐的高优先级客户和待处理事项。
2. 销售进入客户 360，查看客户画像、风险标签、联系时间线。
3. 销售进入商机优先级，看到客户为什么排在前面，比如套餐即将到期、价值等级高、最近有价格异议。
4. 销售把客户上下文带入 Agent 工作台，让 Agent 做客户分析、知识库检索或生成跟进任务建议。
5. 如果 Agent 只是读数据和给建议，直接返回结果；如果要写 CRM，先生成 confirmation。
6. 用户确认后，后端才真正创建任务、写联系记录或更新商机阶段。
7. 管理员可以在运行审计里看到这次 run、tool call、confirmation，便于追踪和排查。
8. 评测报告用 JSONL 用例检查 RAG、Tool Calling、确认覆盖率和延迟指标。

面试时可以补一句：

> 这条链路的核心是“AI 可以建议，但不能绕过业务安全边界直接改 CRM 数据”。

## 4. 每个页面是干什么的

### 今日工作台

定位：销售和主管的首页，不再是技术展示页。

主要看：

- 今日优先客户
- 风险客户
- 待确认 CRM 写操作
- 客户 -> 商机 -> Agent -> 确认 -> 审计 的业务闭环

你要理解：

> 首页是销售每天工作的入口，它把“该跟进谁”和“还有什么写操作没确认”放在最前面。

### 客户 360

定位：看一个客户的完整画像。

主要看：

- 客户基本信息
- 标签
- 风险等级
- 最近联系
- 跟进时间线
- 打开 Agent 分析该客户

你要理解：

> 客户 360 的作用是把分散在 CRM 里的客户信息聚合起来，让销售不用到处找资料。

### 商机优先级

定位：帮销售或主管判断先跟进哪些商机。

主要看：

- 优先级看板
- 评分
- 推荐理由
- 风险和下一步动作
- 打开客户详情或带着商机上下文进入 Agent

你要理解：

> 这里不是黑盒排序，而是可解释排序。每个分数都要能说出原因，比如价值等级、到期时间、联系频率、客户风险等。

### Agent 工作台

定位：销售用自然语言完成分析和建议。

它能做：

- 客户分析
- 知识库检索
- 商机推荐
- 创建跟进任务建议
- 写联系记录建议
- 更新商机阶段建议

你要理解：

> Agent 工作台不是让模型自由发挥，而是让模型在工具列表里选择工具。读工具直接执行，写工具必须走人工确认。

### 知识库问答

定位：销售 SOP 和政策查询。

主要看：

- 用户问题
- 检索结果
- 引用来源
- 低置信拒答

你要理解：

> RAG 的意义是让回答基于公司知识库，而不是让模型凭空编。答案要带引用，没把握要拒答。

### 呼叫中心

定位：处理销售通话文本。

主要看：

- 通话摘要
- 异议识别
- 质检风险
- 生成联系记录确认

你要理解：

> 呼叫中心页模拟销售通话后的处理，把“听完电话 -> 摘要 -> 质检 -> 写联系记录”变成流程。

### 运行审计

定位：管理员排查 Agent 行为。

主要看：

- Agent run
- Tool call
- 输入输出
- 状态
- trace id
- confirmation

你要理解：

> Agent 不能是黑盒。每次模型选了什么工具、工具输入是什么、最后有没有写 CRM，都要能查。

### 系统能力

定位：管理员和产品评审使用的“系统运行中心”。

主要看：

- RBAC 用户、角色和权限是否来自数据库
- 接口限流是否开启，Agent Chat / Model Chat / 普通 API 的限流桶是多少
- Outbox 当前有多少待分发、分发中、死信事件
- 数据保留策略是否开启，哪些审计/日志数据超过保留周期，哪些数据因为待处理状态被保护
- Tool Calling schema 是否完整，哪些工具是写工具、需要确认
- 模型、Embedding、pgvector 向量库是否接通

你要理解：

> 系统能力页不是给销售日常用的，它把技术能力收进系统管理区。管理员可以用它判断系统是否处于可运行、可追踪、可恢复、可治理的状态。

### 质量评估

定位：验证系统效果。

主要看：

- RAG Recall@5
- Citation Hit Rate
- Refusal Accuracy
- Tool Calling Success Rate
- Write Confirmation Coverage
- 平均延迟和 P95 延迟

你要理解：

> 评测不是为了显示 100 分，而是为了证明 Agent 行为可以被重复验证。

## 5. 后端模块怎么理解

### CRM Core

管理销售业务数据：

- 客户
- 商机
- 联系记录
- 跟进任务
- 产品套餐

这是业务底座。

### Lead Scoring

给商机打分并解释原因。

你可以这样讲：

> 我没有一上来做复杂机器学习模型，而是先做可解释规则评分。因为销售场景里，主管更关心“为什么这个客户排第一”，不是只看一个黑盒分数。

### RAG

让系统从知识库里找依据。

组成：

- 文档切分
- embedding 向量化
- pgvector 存储和检索
- 关键词 + 向量混合检索
- 引用生成
- 低置信拒答

你可以这样讲：

> RAG 解决的是“模型回答要有依据”。知识库内容先切成 chunk，再用 embedding 模型转成向量，存到 pgvector。查询时同时看关键词和语义相似度，最后把引用带回前端。

### Agent Orchestrator

这是 Agent 主流程控制器。

它负责：

- 接收用户消息
- 让 LLM 根据工具 schema 选择工具
- 工具调用失败时回退规则路由
- 记录 agent_run
- 记录 agent_tool_call
- 写操作生成 confirmation

你可以这样讲：

> Orchestrator 相当于 Agent 的调度中心。模型负责理解用户意图和选择工具，但真正执行工具、校验权限、写确认记录，是后端服务控制的。

### Tool Registry

工具注册表。

它定义：

- 有哪些工具
- 工具是读还是写
- 是否需要人工确认
- 参数 schema 是什么

你可以这样讲：

> Tool Registry 是为了让工具调用可控。不是模型想调什么就调什么，而是后端把允许的工具和参数 schema 明确暴露给模型。

### Confirmation Flow

写 CRM 的安全门。

典型流程：

1. Agent 生成写操作建议。
2. 后端保存 confirmation。
3. 前端显示确认卡片。
4. 用户确认。
5. 后端才真正写 CRM。
6. 写入和事件进入审计链路。

你可以这样讲：

> 这是项目里最重要的安全设计。Agent 可以建议写入，但不能直接写入。

### Rate Limit

接口限流。

它解决的问题：

- Agent Chat 和 Model Chat 会调用大模型，成本比普通接口高
- 用户连点、脚本刷接口、恶意请求都可能把模型费用和数据库连接打爆
- 面试现场或本地 Demo 也需要避免一次误操作把服务打挂

当前项目怎么做：

- 普通 API、Agent Chat、Model Chat 使用不同的 token bucket
- Agent Chat 和 Model Chat 的限流更严格，因为它们成本更高、耗时更长
- 超过限流后返回 HTTP 429，让前端或调用方稍后重试

你可以这样讲：

> 限流不是为了限制正常用户，而是保护高成本链路。Agent 接口背后可能调 LLM、查 RAG、写审计表，所以我把普通 API、Agent Chat、Model Chat 分成不同限流桶，避免模型费用、数据库连接和线程资源被打爆。

### Outbox Event

事件可靠性机制。

你要讲得准确：

> CRM 写操作事件在业务事务内写入 outbox 表，事务提交后再异步分发。Agent run 和 tool call 这类审计事件也走 outbox 做 at-least-once 分发，但不是所有事件都等价于严格事务一致。

不要说：

> 所有事件都绝对一致、绝不重复。

更专业的说法是：

> 当前是 at-least-once，消费方需要按 eventId 做幂等。生产化可以继续升级到 Debezium CDC。

为什么 Outbox 要 CAS：

- CRM 写操作提交后，afterCommit 会立即尝试分发事件
- 后台定时任务也会扫描 PENDING / FAILED 事件做补偿分发
- 如果两个 worker 同时看到同一条 PENDING 事件，就可能重复发送
- CAS 的作用是先把事件从 PENDING 抢占成 DISPATCHING，只有抢到锁的 worker 能发送
- 如果 worker 崩了，锁超时后可以重新释放，超过最大重试次数进入 DEAD_LETTER

面试时不要说“Outbox 保证绝不重复”。更准确是：

> Outbox CAS 解决的是“多个分发器同时抢同一条事件”的并发问题，把重复发送概率压低，并让失败事件可重试、可死信。整体语义仍然是 at-least-once，所以消费端还要按 eventId 幂等。

### Spring Security

接口保护。

当前项目有两种 token：

- demo token：本地演示兜底用。没有真实用户体系时，把请求映射到默认 demo 用户和默认销售。
- RBAC token：生产化方向。token 不明文保存，只保存 SHA-256 哈希；认证后从数据库加载 userId、salesRepId、角色和权限。

你可以这样讲：

> demo token 是为了本地快速演示；RBAC token 才是更接近真实业务的方案。RBAC 会把 token 映射成数据库用户，再拿到 salesRepId 和权限集合，这样 A 销售只能看自己的客户、商机、Agent run 和 confirmation，避免数据串看。

### Retention 数据保留

数据保留解决的是“系统运行久了以后，审计和日志表会不会无限增长”的问题。

它保护三类数据：

- Agent 运行审计：run、tool call、confirmation
- 知识检索日志：retrieval log
- Outbox 已发布事件

你要讲得准确：

> 保留策略不是简单定时删表，而是先 dry-run 预估影响，再清理超过保留周期、且已经结束的数据。待处理 confirmation、失败/死信 outbox 事件不会被自动删除，避免把还需要人工处理的业务动作清掉。

为什么默认不直接开启：

- 清理是破坏性操作，必须先有数据库备份和恢复演练
- 管理员需要先看 dry-run 结果，确认会删哪些类别、多少行
- 单次清理有最大行数上限，避免一次事务锁住太多历史数据

你可以这样讲：

> 这属于商业化上线才会重视的运维能力。Demo 项目通常只会写审计表，但真实系统还要考虑数据生命周期：哪些审计留 180 天，哪些检索日志留 90 天，哪些 outbox 已发布事件留 30 天，哪些待处理数据不能删。这个项目已经把它放到系统管理页里做成可观察、可预演、可执行的治理流程。

## 6. 几个术语用人话解释

### Dashboard

Dashboard 就是首页工作台或看板。它把最重要的信息放在一个页面上，让用户一打开系统就知道今天该做什么。

### Swagger

Swagger 是接口文档页面。后端写好接口后，Swagger 可以展示接口地址、参数、返回值，还可以在线调接口。

### Actuator

Actuator 是 Spring Boot 的健康检查工具。比如 `/actuator/health` 可以看服务是否正常。

### X-Trace-Id

X-Trace-Id 是一次请求的追踪编号。前端、后端、日志里都带同一个编号，出了问题可以顺着这个编号排查。

### JUnit

JUnit 是 Java 单元测试框架。项目里用它验证后端核心逻辑，比如 Agent 流程、Security、RAG、评分等。

### Playwright

Playwright 是前端自动化测试工具。它像真人一样打开浏览器，点击页面、检查内容，用来验证完整业务流程有没有断。

### Embedding

Embedding 是把文本转成数字向量。相似文本的向量距离更近，这样系统就能做语义检索。

### pgvector

pgvector 是 PostgreSQL 的向量检索扩展。它让数据库可以存储和检索 embedding 向量。

### Tool Calling

Tool Calling 是让模型不要只生成文字，而是从后端提供的工具列表里选择一个工具，并生成结构化参数。

### Rate Limit

Rate Limit 就是接口限流。简单说：一分钟内最多允许某类接口被调用多少次。它保护的是模型费用、数据库连接、后端线程和线上服务稳定性。

### RBAC

RBAC 是基于角色的权限控制。用户不是直接拥有一堆散权限，而是绑定角色，角色再绑定权限。比如销售、主管、系统管理员拥有的页面和接口权限不一样。

### CAS

CAS 是“比较并更新”。在 Outbox 里可以理解成：只有当事件状态还是 PENDING 时，才能把它抢占成 DISPATCHING；如果别人已经抢走了，这次更新就失败。它用来解决并发抢同一条事件的问题。

### RAG

RAG 是检索增强生成。简单说就是：先从知识库找资料，再让模型基于资料回答。

### Retention

Retention 是数据保留策略。简单说：历史审计和日志保存多久，超过周期以后怎么安全清理。

## 7. 面试 3 分钟讲解模板

你可以这样讲：

> 这个项目叫 CRM-AgentPilot，是一个面向本地生活销售场景的 CRM AI Agent 工作台。  
> 我没有把它做成普通聊天机器人，而是围绕销售真实作业流来设计：销售先看今日高优先级客户，再看客户 360 和商机评分，之后进入 Agent 工作台做客户分析、知识库检索或生成跟进任务。  
>   
> 后端核心是 Agent Orchestrator + Tool Registry。模型通过 Tool Calling 在工具 schema 里选择工具，比如客户分析、知识库检索、商机推荐、创建任务。读工具可以直接执行，但写 CRM 的工具必须先生成 confirmation，经用户确认后才真正落库。这样可以避免模型幻觉直接修改业务数据。  
>   
> RAG 方面，我接了阿里百炼 text-embedding-v4，把知识库 chunk 转成 1024 维向量存到 PostgreSQL pgvector，再做关键词和向量混合检索，返回答案时带引用，低置信时拒答。  
>   
> 工程化上，我做了 Spring Security、RBAC token、接口限流、Swagger、Actuator、X-Trace-Id、Agent run/tool call/confirmation 审计表、Outbox CAS 分发与死信、JUnit 后端测试和 Playwright 前端链路测试。  
>   
> 这个项目目前已经从展示型 Demo 推进到产品化 Beta：具备真实 RBAC、限流、Outbox 死信、pgvector 检索、数据保留策略、启动前生产检查和完整链路测试。继续商业化上线时，重点是接企业 SSO、tenantId 多租户、监控告警、数据备份和更大规模真实评测集。

## 8. 产品验收演示顺序

按这个顺序演示最稳：

1. 今日工作台：说明首页服务销售和主管，看今日优先客户、待确认事项和业务闭环。
2. 客户 360：打开一个客户，看标签、风险、时间线。
3. 商机优先级：看该客户的商机分数和推荐理由。
4. Agent 工作台：让 Agent 分析客户，展示工具调用和结构化结果。
5. 创建任务：输入“帮我创建明天上午 10 点跟进美家房产的任务”，展示 confirmation。
6. 确认写入：点击确认后说明任务才真正落 CRM。
7. 运行审计：查看 run 和 tool call，说明可追溯。
8. 质量评估：运行评测或展示指标，说明系统可以被验证。

### 验收前先做数据复位

因为真实验收会不断创建 Agent run、确认单和 CRM 跟进任务，多跑几次以后页面右侧可能堆出很多历史 pending 卡片。验收前建议先复位测试数据，让客户、商机、知识库保留，但把操作过程中生成的数据清干净。

在 F 盘项目目录执行：

```powershell
cd "F:\后端开发新项目\crm-agentpilot"
.\scripts\reset-demo-state.ps1
```

这个脚本会清理：

- Agent 会话、运行记录、工具调用、确认单
- Outbox 事件
- Agent 创建的 CRM 跟进任务
- Agent 写入的联系记录

它不会清理：

- 种子客户
- 种子商机
- 销售知识库
- RBAC 用户和角色

## 9. 产品评审常问问题怎么答

### Q1：这是真 Agent 吗？

答：

> 是一个受控的 CRM Agent，不是开放式聊天机器人。它支持 LLM Tool Calling，模型可以根据 schema 选择工具；但工具执行、写操作确认、审计记录都由后端控制。

### Q2：为什么写 CRM 前要确认？

答：

> 因为模型可能理解错客户、时间或业务意图。如果让模型直接写 CRM，会有数据污染风险。所以我把写工具统一放到 confirmation 流程里，用户确认后才落库。

### Q3：RAG 有什么作用？

答：

> RAG 让回答基于企业知识库，而不是模型凭空生成。项目里通过 embedding + pgvector 做语义检索，再结合关键词得分，最后返回引用来源和低置信拒答。

### Q4：为什么还要规则回退？

答：

> 产品化系统需要稳定性。LLM 调用可能失败、超时或者返回不合法参数，所以保留规则路由作为 fallback，保证核心销售流程不断。

### Q5：Outbox 是什么？

答：

> Outbox 是把业务事件先写进数据库表，再异步发出去。这样 CRM 写入成功后，事件不会只存在内存里，失败也可以重试。为了避免 afterCommit 和定时补偿任务同时发同一条事件，我加了 CAS 抢占：先把事件从 PENDING 更新成 DISPATCHING，抢到锁的 worker 才能发。当前语义是 at-least-once，消费方仍然要按 eventId 去重。

### Q6：现在离生产还有什么差距？

答：

> 主要是五点：企业 SSO/JWT、tenantId 多租户权限、Outbox 下游消费端幂等、真实业务样本评测集、线上监控告警和备份恢复。

### Q7：为什么要做接口限流？

答：

> 因为 Agent Chat 和 Model Chat 是高成本链路，背后可能调 LLM、查知识库、写审计表。没有限流时，用户连点、脚本刷接口或恶意请求都会把模型费用、数据库连接和线程资源打爆。限流后超过阈值直接返回 429，让调用方稍后重试。

### Q8：RBAC token 和 demo token 有什么区别？

答：

> 兼容 token 是本地兜底，方便没有真实用户体系时快速跑起来。RBAC token 是更接近生产的方式：数据库里只存 token 的 SHA-256 哈希，认证后加载 userId、salesRepId、角色和权限。这样不同销售能看到的数据范围不同，不会所有人都用 userId=1。

### Q9：这些生产化能力应该在哪里看？

答：

> 页面上统一放在“系统管理 -> 系统能力”。销售首页只服务销售作业，不再堆技术指标；系统能力页给管理员和产品评审使用，集中展示 RBAC、限流、Outbox、Tool Schema、模型和 pgvector 状态。

## 10. 你需要真正记住的 8 句话

1. 这个项目解决的是“AI 怎么安全进入 CRM 销售作业流”。
2. Agent 可以读数据和给建议，但写 CRM 必须人工确认。
3. Tool Registry 限定模型能调用哪些工具和参数。
4. RAG 让销售话术和政策回答有知识库引用。
5. RBAC token 解决“谁能看谁的数据”，兼容 token 只是本地启动兜底。
6. 接口限流保护高成本 Agent/模型接口，避免费用和数据库被刷爆。
7. Outbox CAS 解决事件并发分发，DLQ 让失败事件可恢复。
8. 当前是产品化 Beta，商业上线还要补企业 SSO、多租户、监控告警、备份恢复和更大评测。
